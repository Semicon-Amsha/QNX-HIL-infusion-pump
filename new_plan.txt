QNX INFUSION PUMP HACKATHON PLAN
Demo-Optimized, Honest, and Technically Grounded

1. WHAT WE ARE BUILDING NOW

This project will be treated as a hackathon/demo system, not as a full medical software implementation.
The goal is to build and demonstrate a reliable Hardware-in-the-Loop infusion pump controller prototype using:

- QNX Neutrino on Raspberry Pi as the real-time controller
- STM32 Nucleo-F401RE as the hardware-side motor/fault simulator
- UART as the physical communication link
- QNX native message passing for internal IPC between QNX processes

We are NOT optimizing for a large or academically perfect architecture.
We are optimizing for a stable demo that can be explained clearly, built in limited time, and defended honestly.

Important honesty rule:
We may reference IEC 62304 ideas such as fault handling, traceability, and safety-oriented behavior, but we must not imply that this project is medically certified or fully compliant. At best, this is a classroom prototype influenced by safety-critical design principles.


2. MAJOR DESIGN DECISION

We will NOT implement a custom QNX resource manager for this project.

Reason:
- A resource manager is more complex than what we need for a hackathon demo.
- It adds implementation overhead without improving the visible demo as much as protocol handling, fault logic, and reliable IPC.
- We already understand QNX message passing better than custom resource manager internals.
- Message passing is fully appropriate for this architecture because the important internal problem is process-to-process coordination, not exposing a POSIX-style device interface.

Replacement approach:
- Use a normal QNX server process for motor control
- Use MsgSend / MsgReceive / MsgReply for synchronous commands
- Use pulses for asynchronous fault notification
- Keep UART ownership inside one QNX server process only

This is still a valid and QNX-native design.


3. REVISED SYSTEM ARCHITECTURE

The architecture will be:

- demo_client
  Small user-facing test application on QNX
  Sends START / STOP / STATUS requests to motor_ctrl using message passing

- motor_ctrl
  Main control server on QNX
  Owns the UART device /dev/ser1
  Builds framed packets
  Sends commands to STM32
  Waits for ACK / STATUS / FAULT responses
  Detects timeout and heartbeat failure
  Sends pulses to alarm_mgr on fault conditions

- alarm_mgr
  QNX process that receives pulses from motor_ctrl
  Logs or prints alarm events
  Can be demonstrated as the fault-handling endpoint

- cpu_hog
  Optional stress process
  Used only if time permits, to demonstrate APS or scheduling robustness

- STM32 firmware
  Simulates the hardware side of the infusion pump
  Receives UART packets
  Executes a simple state machine
  Controls onboard LED
  Sends ACK, STATUS, HEARTBEAT, and FAULT packets


4. WHY MESSAGE PASSING IS THE RIGHT IPC HERE

QNX message passing is appropriate because:

- It is the native QNX mechanism for synchronous request/reply control flows
- Our commands are naturally request/response operations: START, STOP, STATUS
- It allows one process to safely own the serial port while other processes interact with it indirectly
- It keeps the design modular and avoids multiple clients touching /dev/ser1 directly
- It fits fault delivery well when combined with pulses

Internal separation:
- UART is for QNX <-> STM32 communication
- QNX message passing is for QNX process <-> QNX process communication

This is cleaner than letting every application write directly to the UART device.


5. FINAL PROJECT SCOPE

The true core scope of the project is:

- QNX sends framed UART commands to STM32
- STM32 responds with valid framed packets using CRC8
- QNX parses those packets correctly
- QNX exposes a message-based control interface through motor_ctrl
- STM32 state changes are visible on the LED
- Faults are detected and reported to alarm_mgr
- At least one clear safety behavior is demonstrated:
  CRC failure handling, explicit FAULT packet handling, UART timeout handling, or heartbeat loss handling

The project is successful even if we do not finish every originally imagined feature.


6. WHAT WE ARE EXPLICITLY NOT DOING

To protect the schedule, we will not depend on the following for the main demo:

- custom resource manager
- large POSIX device interface such as /dev/pump_motor
- complex multi-client support
- advanced UI
- persistent storage design
- full IEC 62304 lifecycle documentation
- exhaustive timing validation on every requirement
- simultaneous heavy development on STM32 and QNX at the same time

If any of these appear later, they are stretch items only.


7. QNX PROCESS DESIGN

7.1 motor_ctrl

motor_ctrl will be the most important QNX process.
It should be implemented as a message-passing server, not as a resource manager.

Responsibilities:
- create a channel and attach a known name
- open and configure /dev/ser1
- serialize outgoing UART access
- build packets using the finalized protocol
- parse incoming bytes into packets
- apply timeout and heartbeat rules
- reply to client requests
- deliver pulses to alarm_mgr on fault conditions

Recommended QNX APIs:
- name_attach()
- name_open()
- ChannelCreate()
- MsgReceive()
- MsgReply()
- ConnectAttach() if needed
- MsgDeliverEvent()
- SIGEV_PULSE_INIT()

Recommended internal model:
- one main dispatch loop for client messages
- one UART RX thread that blocks on read() and parses packets
- one mutex protecting UART transmit path

This matches the existing reference architecture and keeps UART ownership centralized.

7.2 demo_client

demo_client should stay very small.

Responsibilities:
- connect to motor_ctrl using name_open()
- send one command at a time using MsgSend()
- print result in a simple terminal-friendly way

Supported commands:
- start
- stop
- status

This program replaces the need for a fake shell device interface.

7.3 alarm_mgr

alarm_mgr should be simple and reliable.

Responsibilities:
- create channel and wait for pulses
- print or log which fault occurred
- optionally count faults for the demo

This gives a strong demonstration of asynchronous QNX IPC.


8. RECOMMENDED MESSAGE DEFINITIONS

Normal request/reply should be synchronous.
Fault delivery should be asynchronous.

Suggested request structure:

typedef struct {
    uint16_t type;
    uint16_t seq;
    uint8_t  cmd;
    uint8_t  data_len;
    uint8_t  data[16];
} pump_msg_t;

Suggested reply structure:

typedef struct {
    int32_t  status;
    uint8_t  rsp_cmd;
    uint8_t  data_len;
    uint8_t  data[16];
} pump_reply_t;

Suggested message meanings:
- type = command request type
- seq = optional sequence number for debugging
- cmd = protocol command such as START, STOP, STATUS
- status = 0 for success, negative values for failure

Suggested pulse codes:
- PULSE_FAULT_UART_TIMEOUT
- PULSE_FAULT_HEARTBEAT_MISS
- PULSE_FAULT_BAD_CRC
- PULSE_FAULT_STM32_FAULT

The exact struct names can change, but the design should stay this simple.


9. UART PROTOCOL SCOPE

The packet format remains:

[0xAA][CMD 1B][LEN 1B][DATA nB][CRC8 1B]

Command set:
- 0x01 MOTOR_START
- 0x02 MOTOR_STOP
- 0x03 STATUS_REQ
- 0x04 HEARTBEAT
- 0x05 ACK
- 0x06 FAULT

CRC:
- CRC8-MAXIM (Dallas)
- reflected form using 0x8C in LSB-first implementation
- must match on both QNX and STM32

Core protocol expectations:
- QNX sends START/STOP/STATUS
- STM32 returns ACK for valid commands
- STM32 may return STATUS information
- STM32 periodically sends HEARTBEAT
- STM32 may send FAULT at any time


10. FAULT POLICY

We should implement only the fault rules we can actually demonstrate correctly.

Core faults worth implementing:
- UART timeout waiting for reply
- CRC mismatch on received packet
- explicit STM32 FAULT packet

Recommended next fault:
- heartbeat missing for more than 2 seconds

On fault, motor_ctrl should:
- stop motor logically on the QNX side
- if relevant, send MOTOR_STOP to STM32
- notify alarm_mgr through a pulse
- reply failure to the waiting client, if one exists

For the demo, it is better to implement 3 fault cases cleanly than 7 fault cases poorly.


11. STM32 SIDE EXPECTATIONS

The STM32 firmware should remain simple.

Required behavior:
- receive framed UART packets
- validate CRC
- execute a small state machine
- send ACK or FAULT responses
- answer STATUS requests
- blink or set LED according to state
- optionally use push button to inject a FAULT condition

Recommended visible state mapping:
- LED OFF = IDLE
- LED ON = RUNNING
- LED BLINK = FAULT

This is excellent for a demo because the audience can see the system state without reading logs.


12. REVISED BUILD ORDER

The original slice plan should be adjusted for demo value and implementation risk.

Phase A: Stabilize Slice 2
- finalize protocol.c / protocol.h
- validate packet build + parse on QNX
- confirm STM32 and QNX CRC match
- run slice2_test successfully for START, STOP, STATUS

Exit condition:
- QNX sends framed commands and receives valid framed responses from STM32

Phase B: Replace resource manager slice with message-passing motor_ctrl
- create motor_ctrl server using name_attach
- move UART logic behind the server
- create demo_client using MsgSend
- ensure only motor_ctrl owns /dev/ser1

Exit condition:
- demo_client start/stop/status works through QNX message passing

Phase C: Add alarm_mgr with pulses
- create alarm_mgr process
- register pulse event connection
- send pulses on timeout, CRC error, or explicit FAULT

Exit condition:
- a fault is visible both in motor behavior and in alarm_mgr output

Phase D: Improve demo robustness
- add heartbeat support
- clean terminal logs
- make startup order repeatable

Exit condition:
- one reliable end-to-end demo script works repeatedly

Phase E: Stretch only if time remains
- APS partitioning
- cpu_hog demonstration
- tracelogger timing capture

Exit condition:
- only attempt this after Phases A-D are stable


13. WHAT SHOULD BE THE PRIMARY DEMO

Primary demo should be:

1. QNX motor_ctrl starts and owns UART
2. alarm_mgr starts and waits for fault pulses
3. demo_client sends START
4. STM32 LED turns ON
5. demo_client sends STATUS
6. QNX reports running state
7. user presses STM32 button to inject fault
8. STM32 sends FAULT
9. motor_ctrl detects fault and notifies alarm_mgr
10. LED enters FAULT indication or motor stops

This is strong because it demonstrates:
- real hardware interaction
- framed protocol
- CRC-based communication discipline
- QNX message passing
- asynchronous fault handling
- visible safe-state behavior


14. SECONDARY DEMO OPTION

If heartbeat is implemented reliably, an alternative demo is:

1. START command succeeds
2. normal heartbeat packets are shown
3. UART link is unplugged or STM32 heartbeat stops
4. motor_ctrl detects heartbeat miss
5. alarm_mgr receives pulse
6. system transitions to emergency stop behavior

This is a very good demo, but only if it is stable.
Do not choose it as the main demo if it is still flaky close to presentation time.


15. APS AND TRACELOGGER POSITIONING

APS and tracelogger are valuable, but they should not be allowed to endanger the core demo.

Recommended rule:
- If Phases A-D are not stable, do not spend major time on APS or tracelogger
- If the core demo is stable, add one APS demo only
- If APS is stable, add tracelogger screenshots or one measurement

Honest presentation wording:
- "Adaptive Partitioning and timing analysis are planned extensions and partially demonstrated where time allowed."

Do not claim all timing requirements are validated unless actual measurements were captured.


16. WHAT TO CUT FIRST IF TIME SLIPS

If schedule pressure increases, cut in this order:

1. tracelogger measurements
2. cpu_hog / APS demonstration
3. heartbeat miss detection
4. advanced STATUS payloads
5. any attempt at resource-manager-like features

Do not cut:
- packet framing
- CRC correctness
- message-passing motor_ctrl
- visible LED state behavior
- at least one real fault demo


17. IMPLEMENTATION RULES FOR SPEED AND SAFETY

- Never change STM32 and QNX protocol logic at the same time unless absolutely necessary
- After each working milestone, save a copy or snapshot immediately
- Keep all command IDs and CRC logic in one shared reference
- Log every TX and RX packet during development
- Keep demo_client very small
- Keep alarm_mgr very small
- Put complexity in motor_ctrl only where needed
- Do not add threads unless they solve a real problem

Most likely threading split:
- dispatch thread for MsgReceive
- UART RX thread for parsing incoming bytes

Avoid unnecessary architecture growth.


18. WHAT SUCCESS LOOKS LIKE

This project is successful if we can honestly demonstrate:

- a QNX process architecture with message passing
- a working UART protocol with CRC
- a hardware-side STM32 state machine
- visible actuation state through LED
- QNX-side fault handling using pulses
- a coherent explanation of why this design is appropriate for a safety-oriented embedded prototype

This project does NOT need to look industrial to be successful.
It needs to work, be explainable, and survive the demo.


19. PRESENTATION WORDING TO STAY HONEST

Safe wording:
- "We implemented a QNX-based HIL prototype for infusion pump control."
- "We used QNX native message passing for internal IPC and UART for hardware communication."
- "We prioritized reliable command handling, fault detection, and visible safe-state behavior."
- "A custom resource manager was intentionally not chosen because a message-passing control server was sufficient for the project scope and timeline."
- "The design is inspired by safety-critical thinking, but it is a classroom prototype, not a certified medical device."

Avoid saying:
- "fully compliant"
- "production-ready"
- "medically safe"
- "certified"
- "complete IEC 62304 implementation"


20. FINAL DECISION SUMMARY

The project direction is now:

- no custom resource manager
- yes to QNX message passing
- yes to pulses for fault notification
- yes to demo-first development
- yes to a centralized motor_ctrl server
- yes to a small demo_client
- yes to a small alarm_mgr
- yes to framed UART with CRC as the communication foundation

The technical goal is not architectural ambition.
The technical goal is a stable, convincing, and honest demonstration that we can finish in time.
