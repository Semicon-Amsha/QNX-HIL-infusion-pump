- 4GB RAM, AArch64, Cortex-A72
- Running: PISQUARE QNX Neutrino 8.0.0 (2025/01/16 build)
- Boot: SD card, read-only IFS filesystem
- Network: Static IP `192.168.1.50` (set manually each boot)
- Serial: `/dev/ser1` via `devc-serminiuart` at 115200 baud, `-c500000000 -e -F -u1 0xfe215000,125`

### STM32 Nucleo-F401RE
- ARM Cortex-M4, 84MHz
- UART used: **USART1 on PA9(TX) and PA10(RX)** — NOT USART2 (USART2 is internally routed to ST-Link)
- Baud: 115200, 8N1
- LED: PA5 (LD2 onboard)
- Button: PC13 (occlusion fault injection)
- Development: STM32CubeIDE on Windows PC

### Physical UART Wiring
```
STM32 PA9  (TX) → Pi GPIO15 Pin10 (RX)
STM32 PA10 (RX) → Pi GPIO14 Pin8  (TX)
STM32 GND       → Pi Pin6   (GND)
```
- 3.3V logic, no level shifter needed
- STM32 powered via USB to PC

## Packet Protocol (Finalized)
```
[0xAA][CMD 1B][LEN 1B][DATA nB][CRC8 1B]
```

### Command IDs
| ID | Name | Direction | Description |
|---|---|---|---|
| 0x01 | MOTOR_START | QNX→STM32 | Start motor |
| 0x02 | MOTOR_STOP | QNX→STM32 | Stop motor |
| 0x03 | STATUS_REQ | QNX→STM32 | Request status |
| 0x04 | HEARTBEAT | STM32→QNX | 500ms keepalive |
| 0x05 | ACK | STM32→QNX | Command acknowledged |
| 0x06 | FAULT | STM32→QNX | Fault detected |

### CRC8 Polynomial
- **CRC8-MAXIM (Dallas)** polynomial `0x31`
- Must be identical on both STM32 and QNX sides

### Fault Rule (Universal)
Any of these → immediate MOTOR_STOP + fault pulse to alarm_mgr:
- CRC mismatch
- Unknown command received
- No heartbeat for 2 seconds
- UART response timeout (500ms)
- STM32 sends 0x06 FAULT packet

## QNX Process Architecture
```
motor_ctrl  APS 40% → owns /dev/pump_motor + UART RX/TX
alarm_mgr   APS 20% → receives fault pulses via MsgDeliverEvent
logger      APS 20% → UI + logging
cpu_hog     APS 10% → stress test for APS demonstration
system      APS 10% → kernel + drivers
```

### motor_ctrl Internal Threading
```
motor_ctrl process
├── Dispatch thread  — MsgReceive loop, handles /dev/pump_motor clients
├── UART TX          — called from dispatch thread only (mutex protected)
└── UART RX thread   — blocking read, parses packets, delivers fault pulses
```
- One mutex protects UART fd for TX
- RX thread only reads and parses, never writes to UART directly

### IPC Flow (Normal)
```
App → write("/dev/pump_motor") → MsgSend → motor_ctrl dispatch
→ UART TX → STM32 → ACK → motor_ctrl → MsgReply → App unblocks
```

### IPC Flow (Fault)
```
STM32 FAULT packet → UART RX thread → MsgDeliverEvent(pulse)
→ alarm_mgr MsgReceivePulse → prints/logs → motor stopped
```

## STM32 State Machine
```
IDLE ──START──► RUNNING ──STOP──► IDLE
                   │
                 FAULT
                   │
              EMERGENCY_STOP ──► IDLE
```
- LED PA5: OFF=IDLE, ON=RUNNING, BLINK=FAULT
- Button PC13: injects occlusion fault
- Invalid state transitions → NACK response

## Known Pitfalls (Learned the Hard Way)
1. **USART2 is tied to ST-Link** — always use USART1 (PA9/PA10) for external UART
2. **Mini UART needs explicit baud** — always restart with `-b115200 -c500000000`
3. **`/var` wiped on reboot** — never rely on files there surviving reboot
4. **Background `cat /dev/ser1 &`** — always `slay -f cat` before testing
5. **sshd rexec resets config** — always start with `-o "AuthorizedKeysFile /var/.ssh/authorized_keys"`
6. **`sleep()` segfaults on QNX** — use `delay()` or `VTIME` instead
7. **`echo > /dev/ser1`** — doesn't configure termios, use C program for reliable UART
8. **Loopback test crashed QNX** — never short GPIO14/15 directly
9. **STM32 HAL_UART_Transmit in while loop blocks RX interrupt** — never put blocking TX in main loop
10. **Multiple background cats** — use `jobs` and `slay -f cat` to clean up
